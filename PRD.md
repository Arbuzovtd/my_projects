Product Requirements Document (PRD)Проект: Алгоритмический торговый робот (TradingView + Bybit + Telegram WebApp)Версия: 1.0Статус: Утверждено1. Описание продукта (Product Overview)Продукт представляет собой автоматизированную торговую систему для фьючерсного рынка Bybit (USDT-M).Система использует связку: TradingView (как мозг) + Python-сервер (как руки) + Telegram WebApp (как пульт управления).Бот торгует по стратегии возврата к среднему (Mean Reversion на базе VWAP), адаптированной из существующего Pine Script (ER_MR Strategy v2 SpeedC). Бот работает мультимонетно, начиная с базовых активов: BTC, ETH, SOL.2. Архитектура системы (System Architecture)Система состоит из 4 ключевых компонентов:TradingView (Истоник сигналов): Выполняет расчет индикаторов, логики усреднения и условий выхода. Генерирует JSON-сигналы через Webhook.FastAPI Сервер (Ядро бота): Принимает Webhook от TV, читает локальный файл конфигурации (config.json), валидирует данные и отправляет торговые приказы на биржу.Bybit API (Исполнение): Принимает рыночные ордера через официальную библиотеку pybit.Telegram Bot & WebApp (Интерфейс): Служит для мониторинга позиций, изменения объема (мультипликатора) и управления статусом робота (пауза/старт) без необходимости изменения кода или настроек TV.3. Функциональные требования (Functional Requirements)3.1. TradingView (Модификация Pine Script)Скрипт должен отправлять стандартизированный JSON при открытии, усреднении и закрытии позиций.Логика объема: TV передает только "базовые единицы" (например, qty=1, qty=5), расчеты реального объема в USDT делегируются Python-боту.Типы сигналов:entry: Открытие или усреднение (Long/Short).close_all: Полное закрытие позиции (по SL, TP или времени/FC).3.2. Обработчик Webhook (FastAPI)Роутинг: Эндпоинт /webhook, принимающий POST-запросы от TradingView.Безопасность: Защита вебхука секретным токеном (передается в URL или заголовке), чтобы избежать ложных сигналов от третьих лиц.Умная синхронизация (Smart Sync): Перед исполнением сигнала close_all, бот запрашивает статус позиции на Bybit. Если позиция уже закрыта пользователем вручную, бот игнорирует сигнал TV и не выдает ошибку API.3.3. Торговая логика и управление рискамиМасштабирование объема (Multiplier): При получении сигнала на вход, бот умножает присланный из TV qty на мультипликатор из config.json для конкретной монеты.Failsafe (Обработка нехватки маржи): * Если Bybit отклоняет ордер из-за нехватки средств/маржи (особенно критично при усреднении открытой позиции), бот немедленно отправляет Alert в Telegram.Бот переводит статус монеты в config.json в режим paused_for_entries (пауза новых входов).Критично: В этом режиме бот игнорирует новые сигналы entry по этой монете, но обязан исполнить сигнал close_all, когда он придет из TV, чтобы закрыть зависшую позицию.3.4. Telegram WebApp (Пользовательский интерфейс)Мини-приложение, открывающееся по кнопке в боте.Функционал дашборда:Отображение списка монет (BTC, ETH, SOL).Статус каждой монеты: 🟢 Активен, 🔴 На паузе (с указанием причины).Текущая позиция (Long/Short, объем, PnL в реальном времени с Bybit).Управление: Ползунок или поле ввода для изменения мультипликатора объема (Multiplier).Кнопка «Возобновить ▶️» для снятия монеты с паузы после ошибки маржи.Кнопка «Экстренное закрытие 🛑» (опционально) для закрытия позиции по рынку и остановки бота по монете.4. Стек технологий (Tech Stack)Проект строится на современных, легковесных и быстрых решениях:Пакетный менеджер и сборка: uv (от Astral) — для сверхбыстрого управления зависимостями.Backend / Webhook receiver: Python 3.12+, FastAPI, uvicorn.Биржа: pybit (Официальный SDK Bybit V5).Telegram Bot / WebApp Backend: aiogram (v3.x).Хранилище состояний: Локальный JSON-файл (config.json). Отказ от реляционных БД (SQLite) и In-Memory кэшей (Redis) в пользу максимальной простоты и надежности.Frontend (WebApp): HTML5, CSS3 (возможно использование Tailwind via CDN), Vanilla JavaScript.5. Структура данных (State Management)Файл config.json имеет следующую структуру (пример):{
  "BTCUSDT": {
    "status": "active", 
    "multiplier": 10.0
  },
  "ETHUSDT": {
    "status": "active",
    "multiplier": 5.0
  },
  "SOLUSDT": {
    "status": "paused_for_entries",
    "multiplier": 20.0
  }
}
Где status может быть: active, paused (остановлен полностью), paused_for_entries (остановлен из-за ошибки, ждет закрытия сделки).6. Инфраструктура и развертывание (Deployment)Хостинг: Арендованный VPS-сервер (ОС Ubuntu 22.04+).Контейнеризация: Проект упаковывается в Docker-образ с использованием Dockerfile (базирующемся на образе с установленным uv).Запуск: docker-compose.yml для поднятия контейнера и автоматического перезапуска при падении (restart: unless-stopped).Хранение данных: Файл config.json монтируется через Docker Volume на хост-машину, чтобы настройки не сбрасывались при обновлении контейнера.7. Этапы разработки (Roadmap)Фаза 1: Подготовка TradingView. Модификация исходного Pine Script для формирования и отправки JSON-вебхуков.Фаза 2: Базовый Python-сервер. Инициализация проекта через uv, настройка FastAPI и pybit, реализация получения вебхуков и открытия/закрытия тестовых ордеров.Фаза 3: State Management & Failsafes. Реализация логики чтения/записи config.json, обработка ошибок маржи Bybit, защита от рассинхронизации.Фаза 4: Telegram WebApp. Написание Telegram-бота (aiogram), создание HTML-фронтенда, связь фронтенда с бэкендом (управление ползунками и статусами).Фаза 5: Инфраструктура. Написание Dockerfile, деплой на арендованный сервер, итоговое тестирование.
